# CrossBridge Android

This is the Android app for CrossBridge.

Current scope:
- Kotlin Android app shell
- Jetpack Compose home, camera QR scanner, QR JSON paste, pairing confirmation, and trusted devices screens
- CameraX camera preview with ML Kit barcode scanning for QR codes
- Relay WebSocket pairing client
- CrossBridge QR payload parser
- Android identity store backed by SharedPreferences
- Trusted Windows device store backed by SharedPreferences JSON
- Trusted-device auto-connect while the Android app is open
- In-app text/link sharing with trusted online Windows devices
- Manual Copy, Open, and Android Share actions for received items
- Android-facing relay URL persistence for emulator and physical-phone development
- JVM unit tests for QR parsing, pairing-code parity, trusted-device storage, and identity storage
- File transfer in both directions over the encrypted relay path
- Android notification listener service, metadata mirroring to Windows, and reply/dismiss actions
- Local LAN fast path direct TCP transfer (port 8789) with automatic relay fallback
- Internet, network state, camera, file transfer, and notification listener permissions
- `minSdk` 26
- `targetSdk` 36

Not included yet:
- Foreground service declarations
- Screen mirroring
- Background clipboard sync

Those permissions should be added only when their milestone is implemented.

## Android build environment

Required:

- JDK 17 or newer
- Android Studio or Android SDK command-line tools
- Android SDK platform API 36 (`platforms;android-36`)
- Android SDK build tools 35.0.0 (`build-tools;35.0.0`)
- The committed Gradle wrapper in this folder

Set one Android SDK environment variable to the SDK directory:

```bash
export ANDROID_HOME="$HOME/Android/Sdk"
export ANDROID_SDK_ROOT="$ANDROID_HOME"
```

Windows PowerShell:

```powershell
$env:ANDROID_HOME="$env:LOCALAPPDATA\Android\Sdk"
$env:ANDROID_SDK_ROOT=$env:ANDROID_HOME
```

This repo includes the generated Gradle 8.13 wrapper. Installed Gradle is only needed if the wrapper needs to be regenerated.

Generate the wrapper only with Gradle:

```bash
cd apps/android
gradle wrapper --gradle-version 8.13
```

Do not create a placeholder `gradle-wrapper.jar`; the wrapper JAR must be generated by Gradle.

On Windows, the Android npm scripts also check Android Studio's bundled JBR and the default SDK location when `JAVA_HOME`, `ANDROID_HOME`, or `ANDROID_SDK_ROOT` are not visible in the current shell. Explicit environment variables still take precedence.

The GitHub Actions Android workflow sets up JDK 17, Android SDK platform 36, build tools 35.0.0, and Gradle cache support, then uses the committed real Gradle wrapper.

Android dependency versions are kept in the Gradle files for now: plugin versions live in `build.gradle.kts`, and app library versions live near their dependencies in `app/build.gradle.kts`. Add a version catalog only when the Android project grows enough to need it.

## Build and Test

```bash
npm run doctor
npm run test
npm run build
```

Equivalent Gradle commands from `apps/android`:

```bash
./gradlew :app:testDebugUnitTest
./gradlew :app:assembleDebug
```

On Windows, use `.\gradlew.bat`.

Windows PowerShell equivalents from this folder:

```powershell
.\gradlew.bat :app:testDebugUnitTest
.\gradlew.bat :app:assembleDebug
```

The debug APK is produced at:

```text
apps/android/app/build/outputs/apk/debug/app-debug.apk
```

Android verification is separate from the root `npm run check` command for now so TypeScript protocol, relay, and Windows frontend checks can run on machines without Java, Gradle, or the Android SDK.

Manual development pairing:

1. Start the relay with `npm run dev:relay`.
2. Start the Windows UI with `npm run dev:windows`.
3. Open the Windows Pair page.
4. Keep `Windows relay URL` as `ws://127.0.0.1:8787/connect`.
5. Set `Android QR relay URL`:
   - Emulator: `ws://10.0.2.2:8787/connect`
   - Physical phone: `ws://<PC-LAN-IP>:8787/connect`
6. Create a pairing code on the Windows Pair page.
7. Copy the QR payload JSON.
8. Open the Android app and tap `Scan QR code` to scan the QR code on the Windows Pair page. Or tap `Paste QR JSON for dev testing` for the dev fallback.
9. Paste the JSON, join, compare the 6-digit code, and confirm on both devices.
10. Open Android Trusted devices and Windows Devices to confirm both sides saved the trusted device.
11. Restart Android and refresh Windows to confirm trusted-device persistence.
12. Keep the relay running and confirm both apps reconnect automatically without another QR scan.

VPN can stay on. CrossBridge pairs through encrypted relay mode.

Manual development text/link sharing:

1. Start the relay with `npm run dev:relay`.
2. Start the Windows UI with `npm run dev:windows`.
3. Install the Android debug app with `.\gradlew.bat :app:installDebug`.
4. Pair Windows and Android once, or use already saved trusted devices.
5. Confirm both apps show the trusted peer online.
6. On Android, open `Share text or link`, enter text or a URL, and send to Windows.
7. On Windows, open `Share`, enter text or a URL, and send to Android.
8. Use Copy for received text and Open for received URLs.

What works now:
- Windows to Android text/links.
- Android to Windows text/links.
- File transfer in both directions.
- Notification mirroring, dismiss, and reply.
- Optional local LAN fast path direct TCP socket transfer.
- Automatic and transparent relay fallback when direct local socket is blocked.

What does not work yet:
- Background clipboard sync.
- Screen mirroring.
- Foreground service background reconnect.

Current payload protection:

- Text/link payloads use encrypted trusted relay envelopes.
- Current app-layer crypto uses ECDH P-256, HKDF-SHA256, AES-GCM, a fresh random nonce per message, and authenticated relay metadata.
- This is development-grade real encryption, not production-grade E2EE. The current real ECDH private key is stored in SharedPreferences; Android Keystore-backed protection remains pending.
- The relay must not inspect, permanently store, or log text/link content.

For physical-phone testing, start the relay with `CROSSBRIDGE_RELAY_HOST=0.0.0.0` if the phone cannot reach the PC LAN address while the relay is bound to loopback.

The debug build allows local `ws://` relay URLs so emulator and LAN development pairing can run against the local relay. This is a development runtime pairing and auto-connect flow, not the final installer or APK distribution workflow.

After pairing, Android saves the Android-facing relay URL from the QR payload. Emulator auto-connect normally uses `ws://10.0.2.2:8787/connect`; physical-phone auto-connect normally uses `ws://<PC-LAN-IP>:8787/connect`. Windows auto-connect normally uses `ws://127.0.0.1:8787/connect`.

Current Android identity uses a development ECDH P-256 key pair in SharedPreferences. Android Keystore-backed identity storage and the preferred X25519 agreement are pending. Native camera QR scanning is implemented. Full production background service behavior and the final APK release workflow are not implemented yet.

## Camera Permission

The Android app requests camera permission only when the user opens the QR scanner screen. Camera permission is not requested at app launch or on the home screen. If the user denies camera permission, the QR JSON paste fallback is still available.

The `CAMERA` permission is declared in the manifest with `android.hardware.camera` as `required="false"` so the app can be installed on devices without a camera, such as emulators.
